# 全栈 QA 验收与边缘防御指南 (QA Guidelines)

本指南跳出单纯的“点击测试”，要求研发团队基于状态机模型，严格执行全栈的边缘防线演练。

## 1. 前端交互与状态自愈防线 (Frontend Edge Cases)

前端的本质是正确渲染状态机，并在弱网或接口异常时平滑降级。

### A. 状态不同步与回弹
- **测试场景**：用户点击开启 `Data Training Opt-in` 的瞬间，人为使用 Chrome DevTools 掐断网络（或 Mock 返回 HTTP 500）。
- **验收断言**：UI 上的 Toggle 开关不能处于绿色的“假开启”状态。必须在 Catch 到报错后，平滑回弹至初始的关闭状态，并弹出轻量级 Toast 提示 "Sync Failed"。

### B. 授权失效的隐式捕获
- **测试场景**：在设置页停留时，后台模拟修改 `connections.notion.status` 为 `token_expired`。
- **验收断言**：前端下一次发起的轮询或主动刷新后，必须立刻阻断对该项的配置更改，将其渲染为显眼的橙色 `Re-authenticate` 卡片引导。

---

## 2. 后端并发与原子阻断防线 (Backend Edge Cases)

后端的本质是确保状态机的绝对一致性，绝不能被恶意的客户端请求穿透。

### A. 并发写满的竞态防御 (Storage Race Condition)
- **测试场景**：用户的云空间上限为 10GB，当前已使用 9.99GB（剩余 10MB）。此时，用户使用脚本伪造两个并行的 HTTP 请求，分别试图上传 8MB 的录音块。
- **验收断言**：必须有且仅有一个请求成功，另一个请求必须严格返回 HTTP 403。
- **防御机制要求**：
  系统必须采用原子化的容量预占/提交机制，确保并发上传时不会突破用户容量上限。具体实现可由工程侧决定，例如数据库事务、行锁、Redis 分布式锁或对象存储回调校验。

### B. 定时任务清理安全 (Retention Cron Job)
- **测试场景**：`Cloud Retention Period` 设置为 7 天。在第 7 天的凌晨，系统执行大批量删除。
- **验收断言**：
  1. 不得锁表导致此时用户的其他设置操作（如更改 `default_format`）被卡死。
  2. 哪怕删除任务失败中途 Crash，重跑时必须具备幂等性。

---

## 3. 隐私与默认安全基线 (Privacy Baseline)
- **强制检查项**：无论通过网页端注册，还是硬件激活进入的新用户，数据库中的 `data_training_opt_in` 字段**绝对不得存在默认 true 的代码**。必须显式为 `false` 或 `null`（同等视作 false）。
